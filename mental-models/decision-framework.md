# 决策框架

## 技术决策优先级

1. **性能** — 硬约束，EMS 系统对实时性有刚性要求
2. **安全性** — Rust 编译器兜底，"编译即跑通"
3. **Agent 友好** — 代码结构化、JSON API 合理、CLI 工具链完备（如 monarch CLI）
4. **人的可读性** — 人能读懂、能维护

> AI 辅助开发时代，实现难度不再是否决因素。Rust 的学习曲线被 Claude Code 抹平。

## 项目立项判断

三道关卡，依次过滤：

1. **安全性（一票否决）** — 全维度：技术安全、商业安全、团队安全。任何一个维度有硬伤，直接否
2. **团队覆盖度 ≥ 60%** — 当前团队的技术栈和认知栈能 cover 60% 以上才做，否则风险不可控
3. **第一性原理** — 回到最根本的需求，这件事本身是否合理。不被表面需求带着走

## Tiebreaker

当两个方案都说得通时：

> **选有学习价值的，但有成本天花板。**

能让团队成长的方案优先，除非投入产出比明显不合理。

反面例子：用大模型 fine-tune 功率预测模型，理论可行但投入过大 → 否。
